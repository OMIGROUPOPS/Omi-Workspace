# C-P0-RACE / C-P0-BUILD — cancel/fill race fix (Plex-ratified full scope)

One diff family to `live_v4.py`: every Position-deleting cancel site resolves the
cancel against EXCHANGE truth before any state is dropped. Closes the observed
2026-06-11 05:42:10 ET bug (AUGFUC-AUG fill deleted unbooked by the marketable-stale
cancel race) and its four sibling instances, plus the reconcile-adoption T50 hole.

## STEP 0 — cancel API shape (the gating fact)

**Our DELETE plumbing returns only an HTTP-status boolean.** `_real_api_delete`
(live) returns `r.status in (200, 204)` and discards the body; paper
`handle_delete` likewise returns bare True/False (False when the order isn't
resting — indistinguishable from "already filled"). Kalshi's documented
CancelOrder response (200) carries `{"order": <Order incl. fill counts>,
"reduced_by"}`, but (a) we throw it away, and (b) the failure case we care most
about — already-filled — is an HTTP error with **no fill info at all**. No
observed cancel payload exists in our logs (nothing in the repo ever parsed one;
`reduced_by` appears nowhere).

**Consequence:** resolution ALWAYS requires a get-order poll; all three branches
of `_cancel_entry_and_resolve` decide from that poll. We deliberately did NOT
upgrade `api_delete` to return bodies — it is a shared primitive under every
cancel in the bot (exit consolidation, DCA cleanup, Bug-4-adjacent
settlement_cleanup); changing its contract has a blast radius this diff doesn't
need. Cost: one extra GET per cancel at the swapped sites.

## STEP 2/3 — the helpers

- `_book_v4_entry_fill(tk, pos, filled, fill_price, status, source=None)` —
  idempotent (qty-monotonic per order: same order+count re-call no-ops; later
  partials book incrementally with `new_fills`). **Atomicity (Plex Gate 3), both
  cited and guarded:** the check→write segment contains NO await (asyncio
  single-thread ⇒ no interleave; asserted structurally in the test suite), and
  the awaited tail (exit application / sibling handler / is_taker fetch) is
  serialized per order_id by `self._booking_inflight` — exactly one concurrent
  caller books, others no-op. Unit-level concurrency tests run on the helper
  directly (gather of two simultaneous bookings; instrumented tail).
- `_cancel_entry_and_resolve(tk, pos, label, source)` → `"booked" | "cancelled"
  | "unresolved"`: cancel (result recorded), then ONE get-order poll (1 retry):
  new fill (regardless of cancel ok, incl. partial+cancelled-remainder) → book
  through `_book_v4_entry_fill`; no fill + poll confirms → "cancelled" (caller
  deletes per its existing discipline); poll unavailable → "unresolved" — the
  caller KEEPS the position and retries next cycle. **Never delete on ambiguity.**
  Evidence events: `cancel_fill_race`, `cancel_resolve_unresolved`.
