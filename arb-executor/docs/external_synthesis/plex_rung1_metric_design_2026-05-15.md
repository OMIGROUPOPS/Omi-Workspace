## Plex's synthesis (output, with LaTeX rendering artifacts cleaned to plain text)

Rung 1 should emit a multi-metric decision surface per (cell, threshold), not a scalar winner table: absolute cents and ROI must both be present, every estimate should carry a confidence interval, and the table should span return, risk, risk-adjusted return, capital efficiency, sample quality, and the subset of binary-Greeks analogs that the current corpus can actually support. Rung 1 is pure Problem 1 in the SIMONS_MODE framing—identify where edge exists and how it scales across thresholds—so the producer should stay strategy-evaluation-only and avoid execution claims that require deeper order book or live queue data not present in Rung 0.

### What the operation has computed before vs what's new

The legacy Rung-1 absorbed analyses already computed the sweep skeleton, but not the full Simons-grade surface. exit_sweep_grid computed per-cell ROI and hit rate across exit thresholds, exit_sweep_curves carried aggregated threshold curves, optimal_exits summarized per-cell optimal exit / optimal ROI / hit-at-opt, and exit_sweep_leader_70_74 proved that fine threshold granularity matters inside a single cell. All four were flagged for recomputation because they were settlement-scored, legacy-foundation analyses rather than exit-optimized, foundation-tier outputs built from the new Rung 0 corpus.

What should carry forward directly into v1 from those legacies is: threshold hit rate, mean/EV ROI, threshold curves, per-cell recommended threshold, and threshold-granularity drilldown behavior. What they did not compute, and what A21/A39 now require, is the rest of the full surface: absolute cents and ROI side by side, confidence intervals on every estimate, standard deviation/downside deviation, settlement-loss frequency, risk-adjusted ratios, capital-efficiency rates, sample-quality flags, and the binary-Greeks analog layer.

A second doctrinal shift matters here: SIMONS_MODE says the operation's alpha thesis is path-to-settlement variance capture, not outcome prediction, and Section 4 splits Problem 1 from Problem 2. That means Rung 1 should inherit only the metrics that answer "which cells and thresholds exhibit robust opportunity," while anything needing queue position, deep depth, or live microstructure beyond top-of-book belongs to later producer layers, not this one.

### Greeks recommendation

The cleanest conclusion is that textbook binary delta is not worth emitting as a ranking metric: if contract price is interpreted as probability, dP/dPr(win) ≈ 1 is nearly tautological and offers no cross-cell discrimination. The useful v1 substitute is an execution-delta proxy: the local slope of threshold hit-rate or EV with respect to threshold, measured by adjacent-threshold finite differences within a cell.

The most meaningful theta analog is time decay of opportunity before the market structurally reveals the answer: how quickly reachable EV or hit rate collapses as time to first extreme touch shortens. That is valuable, but it only becomes computable if Rung 0 includes an explicit row-level minutes_to_first_extreme_touch or equivalent, so it should be queued as a one-column upstream addition if not already present.

The strongest conceptual vega analog is sensitivity of EV to realized intramatch volatility, because the whole doctrine says the edge lives in path variance, not final outcome. But current Rung 0 is built around peak-bounce and threshold reachability rather than full path-volatility descriptors, so a proper vega analog needs either an upstream volatility column or a secondary path-summary input; that makes it v2 rather than tonight's v1.

Gamma is weak in textbook form on a bounded binary, but a pragmatic proxy is still useful: curvature of the threshold-opportunity surface, i.e. second differences of hit rate or EV across neighboring thresholds. That is computable from the Rung 1 sweep itself, so it belongs in v1 as a derived shape metric rather than a foundational row field.

Given F33's depth-chain gap, a more operationally valuable "Greek" than textbook vega may be a liquidity sensitivity proxy: how EV and hit-rate vary with top-of-book size or spread proxies available in Rung 0. That is not classical options language, but it is more honest about the data the operation actually has tonight.

### Open questions for the operator

The docs strongly support Wilson intervals for proportions and bootstrap for distributional estimates, but they do not lock bootstrap design details such as resample count, BCa vs percentile, or whether resampling should be at row, match, or day level for path-sensitive metrics. For simple thresholded return metrics, row-level bootstrap is probably enough; for daily throughput or drawdown, day-block or chronology-respecting bootstrap is the more defensible choice.

The spec should also lock the downside convention for Sortino-like metrics: downside relative to zero realized return, relative to threshold shortfall, or relative to the cell's own mean. Zero-return downside is the cleanest and most interpretable first pass because the thresholded sweep is fundamentally about positive excursion capture, but the docs do not force that choice.

A third open point is chronology retention. True drawdown and some throughput metrics need row dates or an active-day denominator; if the current Rung 0 output does not retain sufficient time identifiers, those metrics become either approximate or deferred. The same applies to unique-match counts and path-sensitive theta/vega analogs: if identifiers or time-to-extreme columns are absent, they should be added explicitly now rather than inferred later.

## Chat-side resolution of Plex's open questions (2026-05-15)

These resolutions lock spec-level choices for the Rung 1 producer. All resolutions verified against Rung 0 v1.1 spec Section 5 (docs/rung0_cell_economics_spec.md, committed at 87103d0d) and against the as-built cell_economics.parquet (Rung 0 landing commit 5ca2d89c).

### Resolution 1 — Bootstrap design

**Within-cell metrics** (the vast majority of v1 metrics): row-level bootstrap, n=1000, BCa where computable with percentile fallback. Match-clustering is NOT required at this level because Rung 0 emits one row per N per side, and the two sides of a paired match fall in different price bands (their T-20m prices sum to ~$1). Within a single cell, rows are from different matches.

**Cross-cell aggregations** (e.g., `variance_across_cells_at_threshold`, any metric that aggregates across cells where both sides of a single match contribute): match-clustered resampling at the `event_ticker` level. Rows sharing an event_ticker either all appear in a bootstrap draw or all don't. Rung 0 column 2 `event_ticker` is the clustering key.

Bootstrap method: BCa where the implementation exists and converges; percentile bootstrap as fallback. BCa is more defensible at modest N and asymmetric distributions, which is Rung 1's operating regime.

### Resolution 2 — Sortino downside convention

The downside is realized return at the actual non-hit terminal value under E32's locked no-stop ride-to-settlement model. Plex's "zero-return downside" suggestion is INCORRECT for this corpus — under E32, a non-hit row settles at 1c or 99c, which for a 30c entry is −29c or +69c, not zero.

The terminal value is already in Rung 0 schema as **column 28 `realized_at_settlement` = settlement_value_dollars − t20m_trade_price**. For non-hit rows, this is the realized cents the position actually earned (or lost) by riding to settlement. Sortino downside deviation is computed over the full distribution of realized cents (hit-row realized = thresholded bounce; non-hit-row realized = column 28 value), with negative values contributing to downside.

Implementation note for Rung 1 spec drafter: per row, compute:
- If `peak_bid_bounce_pre_resolution >= threshold`: realized_cents = threshold
- Else: realized_cents = `realized_at_settlement` (the actual ride-to-settlement value, which is negative for losing rides)

Then aggregate the realized_cents distribution per (cell, threshold) for mean, std, downside_std (negative-values-only-std), Sortino.

### Resolution 3 — Chronology retention
