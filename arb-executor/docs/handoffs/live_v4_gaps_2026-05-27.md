# live_v4.py gaps vs new per-band entry-rule schema — 2026-05-27

**Determination: live_v4.py does NOT support the new per-band (discount, post_start, exit_R, tail_policy) structure. STOPPED per task instruction — no code changes attempted.**

Of the four per-band knobs the new entry-rule analysis produces, only one is deployable today:

| knob | new analysis grain | bot today | supported? |
|---|---|---|---|
| **exit_R** | per fine cell-band (28–33/cat) | per-cell via exit parquet `band_exit_X` (price→cell→X) | ✅ **yes** (parquet swap) |
| **discount** | per fine cell-band | per **regime** (9 coarse 10c bands), `bid_offset_cents` in `per_regime_offsets_v2.csv` | ❌ wrong grain + wrong source |
| **post_start (B1–B4)** | per fine cell-band, bucketed | single `placement_minute` per regime (continuous min) | ❌ no bucket concept; per-regime |
| **tail_policy (skip/fallback)** | per band | **hard-coded taker fallback** for all unfilled at T-20m | ❌ no skip; not configurable |

## Exactly what is missing (code changes required to deploy the new rules)

1. **Per-band entry-param source.** The exit parquet is read for `price_low/price_high/band_exit_X` only (`_load_exit_table`). To carry per-band entry params, either (a) extend the exit parquet schema with `discount`, `post_start`, `tail_policy` columns and teach `_load_exit_table` + the placement path to read them per cell, or (b) replace the per-regime `per_regime_offsets_v2.csv` with a per-cell-band entry table (`anchor_low/high`, `discount`, `post_start`, `tail_policy`) and rewrite `_load_entry_table` + the `entry_table[(cat,regime)]` lookup in `routing_tick` (~line 2730–2737) to key on the fine band instead of the 9 regimes.

2. **Discount lookup granularity.** `routing_tick` currently does `regime = regime_lookup(cat, price)` then `entry_table[(cat,regime)]`. Needs a fine-band lookup (price → cell-band → discount). The 9-regime CSV cannot express the 28–33 per-band discounts (multiple bands collapse into one regime → resolution lost). **A CSV/parquet swap alone is insufficient; the lookup keying must change.**

3. **post_start as B1–B4 bucket.** Today placement is a single `placement_minute` gate (`time_to_start <= placement_min*60`). The new rule wants a bucket (B1=T-4h20→T-3h20 … B4=T-1h20→T-20). Minimal change: map each band's `post_start` bucket to its start minute (B1→260, B2→200, B3→140, B4→80) and feed that as `placement_minute`. This is a translation layer, but still requires the per-band entry table (gap #1/#2).

4. **Per-band tail policy (the real code change).** `_v4_manage_resting_inner` STEP 6 (~line 3152–3176) unconditionally crosses at T-20m. To honor `tail_policy=skip`, it must: read the band's policy; if `skip`, **cancel the resting bid at T-20m and take no position** (no taker cross), and record a no-trade. Today there is no skip branch and no policy field on `Position` or in any table.

## Scope estimate
- Gaps #1–#3 are a **table-schema + lookup refactor** (entry table per-band, placement-minute mapping). Contained but touches `_load_entry_table`, `routing_tick` placement (~2722–2810), and the `Position`/regime plumbing.
- Gap #4 is a **behavioral change** to the managed-resting path (add a skip branch) — small but live-money-sensitive (changes whether unfilled legs trade at all).

## What CAN ship now without code changes
- **exit_R only**, via a new `{cat}_adaptive_exit_bands.parquet` (price_low/high/band_exit_X). This delivers the exit-side of the new map; the entry-side (discount/post_start/tail) needs the changes above.
